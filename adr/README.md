# Architecture Decision Records

An ADR captures a single significant technical decision: the context that forced it, the options considered, and why one was chosen. They exist so that six months from now, nobody has to guess why something is the way it is.

Write one when a decision is expensive to reverse or the reasoning won't be obvious from reading the code — e.g. choice of database, messaging pattern, auth strategy, why a service boundary is drawn where it is.

## Adding a new ADR

1. Copy [0001-record-architecture-decisions.md](0001-record-architecture-decisions.md) as a template.
2. Name it `NNNN-short-title.md`, incrementing from the highest existing number.
3. Fill in Context, Decision, and Consequences. Keep it short — an ADR is a record, not an essay.
